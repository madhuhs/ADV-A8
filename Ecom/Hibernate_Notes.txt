Hibernate ORM

 What is Hibernate is an open-source, lightweight Java framework designed to make database interaction for Java applications easy.

 It was invented by Gavin King in 2001 and has since become one of the most widely used persistence frameworks in Java development.

 Hibernate is an ORM (Object Relational Mapping) tool, which means it maps Java objects to relational database tables, eliminating the need for most SQL code.

 Being a non-invasive framework, it does not force developers to extend specific classes or implement particular interfaces, making integration seamless.

 Hibernate is database-independent ‚Äî it supports multiple database systems without requiring code changes.
 It offers automatic table creation, lazy loading, caching, and transaction management, which improves performance and productivity.

 The framework supports HQL (Hibernate Query Language) in addition to SQL, enabling object-oriented queries.
 Developers can build any type of Java application ‚Äî from small desktop apps to large-scale enterprise systems ‚Äî using Hibernate.


Configuration


 Configuration is  Hibernate helper class that holds all the settings and mappings (entities) needed to bootstrap Hibernate.

 one place we can read hibernate.cfg.xml (or programmatic properties) and discover all annotated entity classes before building the SessionFactory.

Key methods and constructor
 
 new Configuration(): creates an empty configuration object (no settings yet).
 
 configure(String resource) / configure():
 Reads settings from a Hibernate config file.
 
 If you pass a path (e.g., "hibernate.cfg.xml"), it loads that specific file; with no args, it looks for hibernate.cfg.xml on the classpath.
 
 Typical contents: JDBC URL/user/pass, dialect, DDL auto, cache settings, etc., plus mapping discovery (annotated entities or HBM XML).

 If the file isn‚Äôt on the classpath or the path is wrong, you‚Äôll get a HibernateException.
 You can also set properties in code: configuration.setProperty("hibernate.show_sql","true").

 SessionFactory (interface)
 SessionFactory is a thread-safe, heavyweight factory for database sessions.
 We usually make one per database for the lifetime of the app.
 
 How you built it
 configuration.buildSessionFactory():
 Compiles the configuration + mappings into an immutable, ready-to-use factory.
 Expensive operation: do it once at startup, reuse everywhere.

 Lifecycle
 Create once (startup), keep as a singleton/bean.
 Call close() on application shutdown to release pools/caches.

 Common mistakes
 Building a SessionFactory per request (very slow, leaks resources).
 Forgetting to close on shutdown.


Session (interface)

A short-lived, not thread-safe object representing a single unit of work with the DB  like a JDBC Connection.
How you get it
sessionFactory.openSession():
Creates a new Session.
Caller is responsible for closing it: session.close().

What you do inside a Session
CRUD: persist, find,update,remove.
Transaction boundaries (begin/commit/rollback).
First-level cache: the Session keeps track of loaded/changed entities until flush/close.

Common mistakes
Sharing a Session across threads.
Leaving a Session open (leaks connections).
Doing work without a transaction.



Annotations
    Core Annotations
 	1.	@Entity
	‚Ä¢	Purpose: Marks a Java class as a persistent entity (mapped to a database table).
	‚Ä¢	Where to use: On a class.
	‚Ä¢	Important: Must have a no-arg constructor and at least one field annotated with @Id.

	2.	@Table
	‚Ä¢	Purpose: Specifies the database table name for the entity.
	‚Ä¢	Where to use: On a class.
	‚Ä¢	Important: If not specified, Hibernate uses the class name as the table name.

	3.	@Id
	‚Ä¢	Purpose: Marks a field as the primary key of the entity.
	‚Ä¢	Where to use: On a field or getter method.

	4.	@GeneratedValue
	‚Ä¢	Purpose: Defines the generation strategy for primary keys.
	‚Ä¢	Where to use: With @Id field.
	‚Ä¢	Important: Strategies include AUTO, IDENTITY, SEQUENCE, TABLE.

	5.	@Column
	‚Ä¢	Purpose: Customizes mapping of a class field to a table column.
	‚Ä¢	Where to use: On a field or getter.
	‚Ä¢	Important: Can define column name, length, nullability, uniqueness, etc.

	6.	@Transient
	‚Ä¢	Purpose: Marks a field that should NOT be persisted in the database.
	‚Ä¢	Where to use: On a field.

Relationships

	7.	@OneToOne
	‚Ä¢	Purpose: Defines a one-to-one relationship between two entities.
	‚Ä¢	Where to use: On an entity field.
	‚Ä¢	Important: Often combined with @JoinColumn or mappedBy.

	8.	@OneToMany
	‚Ä¢	Purpose: Defines a one-to-many relationship (one entity has a collection of another entity).
	‚Ä¢	Where to use: On a collection field (List, Set).
	‚Ä¢	Important: Requires mappedBy on one side or @JoinColumn to define ownership.

	9.	@ManyToOne
	‚Ä¢	Purpose: Defines a many-to-one relationship (many entities refer to one entity).
	‚Ä¢	Where to use: On a field.
	‚Ä¢	Important: Typically used with @JoinColumn.

	10.	@ManyToMany
	‚Ä¢	Purpose: Defines a many-to-many relationship between two entities.
	‚Ä¢	Where to use: On a collection field.
	‚Ä¢	Important: Often uses @JoinTable to define the join table.

	11.	@JoinColumn
	‚Ä¢	Purpose: Specifies the foreign key column for relationships.
	‚Ä¢	Where to use: On the owning side of @OneToOne or @ManyToOne.
	‚Ä¢	Important: Used to map foreign key name explicitly.

	12.	@JoinTable
	‚Ä¢	Purpose: Defines a join table for @ManyToMany or @OneToMany relationships.
	‚Ä¢	Where to use: On a collection field.
	‚Ä¢	Important: Contains details of joinColumns and inverseJoinColumns.

	Advanced Mapping
	13.	@Embeddable
	‚Ä¢	Purpose: Marks a class whose instances are stored as part of an owning entity.
	‚Ä¢	Where to use: On a class.

	14.	@Embedded
	‚Ä¢	Purpose: Used inside an entity to include an embeddable class.
	‚Ä¢	Where to use: On a field.

	15.	@Lob
	‚Ä¢	Purpose: Marks a field as a large object (CLOB or BLOB).
	‚Ä¢	Where to use: On String (CLOB) or byte[] (BLOB).

	16.	@Enumerated
	‚Ä¢	Purpose: Defines how enums are stored (ORDINAL or STRING).
	‚Ä¢	Where to use: On enum fields.

	Core CRUD on managed/detached entities

	1.	Session.persist(entity)

	‚Ä¢	Transitions a new (transient) entity to managed and schedules an INSERT.
	‚Ä¢	Inserting new rows.
	‚Ä¢	Returns: void.
	‚Ä¢	Fails if the entity already has an existing ID (duplicate). Don‚Äôt use for updates.

	2.  Session.merge(entity) Preferred for Update on DETACHED)

	‚Ä¢	Copies data from a detached entity into a managed instance.
	‚Ä¢	Web layer gave you a detached object.
	‚Ä¢	Returns: Managed instance; use this returned object going forward.
	‚Ä¢	Do not keep using the old detached object; it won‚Äôt track more changes.

	3.	Session.remove(entity) (Delete)

	‚Ä¢	Marks a managed entity for deletion (schedules DELETE).
	‚Ä¢	Deleting rows.
	‚Ä¢	Returns: void.
	‚Ä¢	Entity must be managed; if detached, load/find it first or merge() then remove.

	4.	Session.find(EntityClass.class, id) (Read by PK)
	‚Ä¢	Fetches by primary key; returns null if not found.
	‚Ä¢	Simple lookups.
	‚Ä¢	Returns: Managed entity or null.
	‚Ä¢	Honors first-level cache; may not hit DB if already loaded.

	Session utility & state control
	1.	Session.flush()
	‚Ä¢	Forces synchronization of persistence context to DB (executes pending SQL) without ending tx.
	‚Ä¢	You need DB to reflect changes mid-transaction (e.g., for subsequent queries).
	‚Ä¢	Returns: void.
	‚Ä¢	Can trigger constraint errors early; usually rely on commit to flush.

	2.	Session.clear()
	‚Ä¢	Detaches all managed entities from the session.
	‚Ä¢	Long sessions; free memory before loading big result sets.
	‚Ä¢	Returns: void.
	‚Ä¢	After clear, entities are DETACHED; further changes won‚Äôt auto-persist.

	3.	Session.refresh(entity)
	‚Ä¢	Reloads entity state from DB, discarding local changes.
	‚Ä¢	Ensure you see latest DB data (after triggers/other tx).
	‚Ä¢	Returns: void.
	‚Ä¢	Overwrites in-memory changes; use with care.

	4.	Session.contains(entity)
	‚Ä¢	Checks if the instance is currently managed by the session.
	‚Ä¢	You need to know if merge/detach is required.
	‚Ä¢	Returns: boolean.
	‚Ä¢	false for detached or transient entities.

	5.	Session.detach(entity)
	‚Ä¢	Removes a specific entity from the persistence context (makes it DETACHED).
	‚Ä¢	Stop tracking an object, avoid unintended updates.
	‚Ä¢	Returns: void.
	‚Ä¢	Further changes won‚Äôt be flushed unless you merge() later.

	6.	Session.close()
	‚Ä¢	Closes the session and releases JDBC resources.
	‚Ä¢	End of unit of work (always).
	‚Ä¢	Returns: void.
	‚Ä¢	Accessing managed entities after close leaves them DETACHED.

 Relationships in Hibernate

  Hibernate supports different kinds of relationships between entities.
  These map directly to relationships in relational databases.

	1. One-to-One (1:1)
	‚Ä¢	One entity is associated with exactly one other entity.
        Example: A Contact has one Profile Picture.

	‚Ä¢	Annotation: @OneToOne
		Ways to map:
	‚Ä¢	Shared primary key (both tables share the same ID).
	‚Ä¢	Foreign key (one table has a foreign key column referencing the other).
	
	‚Ä¢	Important points:
	‚Ä¢	Use @JoinColumn on the owning side.
	‚Ä¢	Use mappedBy on the inverse side.
	‚Ä¢	Can add cascade = CascadeType.ALL so both entities save/update together.

    2. One-to-Many (1:N)
	‚Ä¢	One entity can have multiple related entities.
        Example: A Contact has many Call Logs.
	
	‚Ä¢	Annotation:@OneToMany
		Ways to map:
	‚Ä¢	Unidirectional: Only one side knows the relationship.
	‚Ä¢	Bidirectional: Both sides know the relationship (mappedBy on the ‚Äúmany‚Äù side).
	
	‚Ä¢	Important points:
	‚Ä¢	Typically represented by a List or Set.
	‚Ä¢	Needs either a @JoinColumn (unidirectional) or mappedBy (bidirectional).
	‚Ä¢	Use orphanRemoval = true if you want children deleted when removed from the collection.

	3. Many-to-One (N:1)
	‚Ä¢  Many entities refer to one parent entity.
       Example: Many Orders belong to one Customer.
	
	‚Ä¢	Annotation:@ManyToOne
		Ways to map:
	‚Ä¢	Usually the owning side of a @OneToMany relationship.
	‚Ä¢	Always uses @JoinColumn to define the foreign key column.

	‚Ä¢	Important points:
	‚Ä¢	This is the most common relationship (foreign key in child table).
	‚Ä¢	Use fetch = FetchType.LAZY for performance.


	4. Many-to-Many (N:M)
	‚Ä¢	Many entities are related to many other entities.
        Example: A Student can enroll in many Courses, and a Course can have many Students.
	‚Ä¢	Annotation:@ManyToMany

	    Ways to map:
	‚Ä¢	Needs a join table with two foreign keys.
	‚Ä¢	Use @JoinTable to define the join table and its columns.

	‚Ä¢	Important points:
	‚Ä¢	Use mappedBy for the inverse side in bidirectional relationships.
	‚Ä¢	Best to manage associations on one side (to avoid duplicate insert issues).
	‚Ä¢	Use Set instead of List to prevent duplicates.


	Always decide:
	‚Ä¢	Who owns the relationship? (Owning side = defines @JoinColumn or @JoinTable)
	‚Ä¢	Do you want unidirectional (simpler, but fewer navigations) or bidirectional (more natural, but more complex)?


	Cascading & collections helpers (patterns, not methods)
	1.	cascade = CascadeType.ALL (on association)
	‚Ä¢	Propagates persist/merge/remove/refresh to children.
	‚Ä¢	Parent ‚Äúowns‚Äù lifecycle (e.g., Contact ‚Üí CallLogs).

	2.	orphanRemoval = true (on @OneToMany)
	‚Ä¢	Removing a child from the collection schedules a DELETE.
	‚Ä¢	Use it when Child has no meaning without parent.

	üîπ What is Fetch Type?
	‚Ä¢	Fetch Type in Hibernate defines how related entities are loaded from the database.
	‚Ä¢	It decides when associated entities (in relationships like OneToOne, OneToMany, etc.) are fetched:
	‚Ä¢	Eager Loading ‚Üí Loads data immediately.
	‚Ä¢	Lazy Loading ‚Üí Loads data only when needed.

	üîπ Types of Fetching

1. FetchType.EAGER
	‚Ä¢	Loads the associated entity or collection at the same time as the parent entity.
	‚Ä¢	When you query a parent, Hibernate executes JOINs or extra queries to fetch the related data.
	
	‚Ä¢	Pros: Simple to use, ensures data is available.
	‚Ä¢	Cons: Can cause performance issues if associations are large or rarely used.
	‚Ä¢	
	    Default:
	‚Ä¢	@OneToOne ‚Üí EAGER
	‚Ä¢	@ManyToOne ‚Üí EAGER

	2. FetchType.LAZY
	‚Ä¢	Loads the associated entity only when accessed in code.
	‚Ä¢	Uses proxy objects (dynamic loading). Actual SQL query runs only when getter method is called.
	
	‚Ä¢	Pros: Improves performance, avoids unnecessary queries.
	‚Ä¢	Cons: May throw LazyInitializationException if session is closed before access.
	
	‚Ä¢	Default:
	‚Ä¢	@OneToMany ‚Üí LAZY
	‚Ä¢	@ManyToMany ‚Üí LAZY

    Best Practices
	1.	Use LAZY by default to avoid heavy queries.
	2.	Override to EAGER only if you always need the related entity.
	3.	For large collections, always prefer LAZY + batch fetching.

    ‚Ä¢	EAGER = ‚ÄúLoad now, even if not needed.‚Äù
	‚Ä¢	LAZY = ‚ÄúLoad later, only if needed.‚Äù

    	‚Ä¢	Hibernate assigns primary key if needed.
    	‚Ä¢	Object becomes managed and enters the persistence context.

    2. Persistent ‚Üí Detached
    	‚Ä¢	When session closes or object is evicted/cleared from session.
    	‚Ä¢	Hibernate stops tracking changes.
    	‚Ä¢	Object still exists in memory with an ID.
    3. Detached ‚Üí Persistent
    	‚Ä¢	session.merge(obj) ‚Üí copies state from detached object into a managed one.
    	‚Ä¢	session.update(obj) ‚Üí reattaches the object as is.
    	‚Ä¢	merge vs update:
    	‚Ä¢	merge() ‚Üí creates/returns a new persistent copy.
    	‚Ä¢	update() ‚Üí directly reattaches the same object.
    4. Persistent ‚Üí Removed
    	‚Ä¢	Happens via session.remove(obj) or session.delete(obj).
    	‚Ä¢	On commit, Hibernate deletes the row in DB.

   What is HQL?
   	‚Ä¢	HQL stands for Hibernate Query Language.
   	‚Ä¢	It is object-oriented and similar to SQL but works with Java classes and objects instead of database tables and columns.
   	‚Ä¢	Example: Instead of writing SELECT * FROM contact, you write:


   	 Key Features of HQL
    	1.	Object-Oriented ‚Äì Uses class names & property names, not table/column names.
    	2.	Portable ‚Äì Works across databases (SQL dialect handled by Hibernate).
    	3.	Supports Relationships ‚Äì Can navigate associations (@OneToOne, @OneToMany, etc.).
    	4.	Powerful ‚Äì Supports joins, group by, aggregate functions, etc.


     Basic Select
     Query query = session.createQuery("FROM EntityClassName");
     List<EntityClassName> data = query.list();

     Selecting Specific Fields
     Query query = session.createQuery("SELECT c.name, c.email FROM EntityClassName c");
     List<Object[]> results = query.list();

     Where Clause
     Query query = session.createQuery("FROM EntityClassName c WHERE c.name = :name");
     query.setParameter("name", "value");
     List<EntityClassName> results = query.list();

     Joins

     a) Inner Join
     Query query = session.createQuery("SELECT c.memberName, p.memberName FROM EntityClassName1 c INNER JOIN c.EntityClassName2 p");
     List<Object[]> results = query.list();

     b) Left Join
     Query query = session.createQuery(
         "SELECT c.memberName, cl.memberName FROM EntityClassName1 c LEFT JOIN c.EntityClassName2 cl"
     );

     Order By
     Query query = session.createQuery("FROM EntityClassName c ORDER BY c.name ASC");

     Aggregate Functions
     Query query = session.createQuery("SELECT COUNT(c) FROM EntityClassName c");
     Long count = (Long) query.uniqueResult();

     Update
     Query query = session.createQuery("UPDATE EntityClassName c SET c.memberName = :memberName WHERE c.id = :id");
     query.setParameter("memberName", "value");
     query.setParameter("id", 5);
     int rows = query.executeUpdate();

     Delete
     Query query = session.createQuery("DELETE FROM EntityClassName c WHERE c.id = :id");
     query.setParameter("id", 10);
     int rows = query.executeUpdate();

     Criteria API vs HQL
     	‚Ä¢	HQL = text-based query language.
     	‚Ä¢	Criteria API = programmatic query building.
     	‚Ä¢	Example in Criteria:

     CriteriaBuilder cb = session.getCriteriaBuilder();
     CriteriaQuery<ContactEntity> cq = cb.createQuery(ContactEntity.class);
     Root<ContactEntity> root = cq.from(ContactEntity.class);
     cq.select(root).where(cb.equal(root.get("name"), "John"));
     List<ContactEntity> results = session.createQuery(cq).getResultList();
